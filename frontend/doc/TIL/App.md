# App()
- 프론트 앱의 최상위 조립 파일
- 앱 전체에서 공통 기능을 깔고, URL별로 페이지를 연결하며, 로그인 필요한 페이지는 ProtectedRoute로 막는 최상위 라우팅 파일
- 앱 공통 기능을 먼저 깔고, 주소별로 페이지를 매칭
- 기능 : 전역 기능 감싸기, 라우터 열기, URL별 페이지 연결

## import 분석
### 라우터 관련
`BrowserRouter` : 웹 브라우저 주소창 기준으로 라우팅 관리(이제부터 URL 기준 페이지 이동 가능)
`Routes` : 여러개의 Route를 감싸서 라우팅 관리(여러 Route 묶는 박스)(주소별 페이지 규칙 정의)
`Route` : 특정 경로에 특정 페이지 연결(URL과 컴포넌트를 연결)
`Navigate` : URL이 일치하지 않을 때 이동할 페이지 이동(다른 경로로 강제 이동)

### 전역상태/기능 관련
`AuthProvider` : 로그인 상태를 앱 전체에서 공유하게 해주는 껍데기(앱 전체에서 로그인 상태 사용 가능하게 함)
`ToastProvider` : 화면에 토스트 메시지를 띄워주는 껍데기(앱 전체에서 토스트 사용 가능하게 함)
`ProtectedRoute` : 로그인한 사람만 접근할 수 있는 페이지를 감싸는 껍데기(인증 필요한 페이지와 아닌 페이지를 구분하는 핵심)

### 페이지 import 관련
- `LoginPage` : 로그인 페이지
- `SignupPage` : 회원가입 페이지
- `HomePage` : 메인 페이지(대시보드)
- `MyPage` : 마이 페이지
- `DogListPage` : 강아지 목록 페이지
- `DogFormPage` : 강아지 폼 페이지
- `CareRecordListPage` : 케어 기록 목록 페이지
- `CareRecordFormPage` : 케어 기록 폼 페이지
- `CareRecordDetailPage` : 케어 기록 상세 페이지
- `ScheduleListPage` : 스케줄 목록 페이지
- `ScheduleFormPage` : 스케줄 폼 페이지
- `ScheduleDetailPage` : 스케줄 상세 페이지


## 공개페이지, 보호페이지 구분
`<ProtectedRoute>` 해당 컴포넌트 안에 감싸진 페이지는 로그인한 사람만 접근 가능
- LoginPage(로그인), SignupPage(회원가입) 페이지만 보호페이지가 아님(로그인 안해도 접근 가능)
- 나머지 페이지는 모두 보호페이지

## :id
- URL 파라미터

## 라우팅
- 특정 URL에 특정 페이지를 연결하는 방식
``` typescript
<Route path="/" element={<HomePage />} />
```

## 보호라우트
- 페이지를 바로 노출하지 않고 한 번 감싸서 검사하는 구조
``` typescript
<Route path="/" element={<ProtectedRoute><HomePage /></ProtectedRoute>} />
```

## 리다이렉트
- 정의되지 않은 주소로 접속하면 자동으로 이동시킬 페이지
``` typescript
<Route path="*" element={<Navigate to="/" replace />} /> // 홈으로 이동
```
- * : 모든 예외 경로를 의미
- Navigate : 강제이동
- replace : 히스토리를 남기지 않고 이동(뒤로가기 안됨)

### 라우팅(routing)
- 브라우저 주소(URL)에 따라 어떤 페이지 컴포넌트를 보여줄지 정하는 체계
- 리액트 라우팅은 URL 보고 어떤 컴포넌트를 보여줄지 고르는 시스템.
- BrowserRouter가 그 기반을 깔고, Routes 안에서 Route들이 매칭되는 구조

#### 라우트(Route)
- 개별 경로 하나 의미(/~~path)[ex. /dogs, /care-records]
- path : 경로
- element : 해당 경로로 접속했을 때 보여줄 컴포넌트
- 주소 1개 + 보여줄 페이지 1개의 짝

#### 라우팅(Routing)
- 어떤 주소일 때 어떤 페이지를 보여줄지 결정하는 전체 과정
- 주소를 보고 화면을 골라주는 동작 전체가 라우팅

#### Route vs Routing
- `Route` : 규칙 하나(주소 하나 + 페이지 하나)
- `Routing` : 규칙 전체를 적용하여 페이지를 연결하는 전체 시스템

### BrowserRouter
- 리액트 앱에서 브라우저 주소(URL)를 감시하고 관리하는 큰 바깥 껍데기
- 이 앱은 주소창을 기준으로 화면을 바꾸는 라우팅 방식을 사용
- BrowserRouter로 감싸야만 URL이 바뀌었을 때 화면이 바뀜
- 브라우저의 주소창과 히스토리 기능을 사용
- BrowserRouter는 그걸 활용해서
리액트 앱을 진짜 여러 페이지처럼 보이게 만들어줌
- 실제로는 완전히 새 HTML 페이지를 계속 불러오는 게 아니라,
주소는 바뀌었지만 앱 내부에서 컴포넌트만 갈아끼우는 방식
- 브라우저 URL을 기반으로 리액트 라우팅이 동작하도록 해주는 최상위 라우터 컴포넌트
- BrowserRouter가 있어서 /path 주소를 리액트 앱의 페이지 전환 규칙으로 쓸 수 있다
- 브라우저 주소를 리액트 앱이 읽고, 그 주소에 맞는 컴포넌트를 보여주게 만드는 장치

### <a href> 차이
- `<a href>` : 브라우저가 서버에 새 페이지를 다시 요청
- 리액트 SPA : 페이지 전체를 새로고침하지 않고,
앱 내부에서 필요한 화면만 바꾸고 싶어함
- `리액트 라우팅` : 전체 페이지 재로딩 최소화, 앱처럼 부드럽게 화면 전환, 주소 기반 탐색 유지


### SPA(Single Page Application)
- HTML 페이지는 크게 하나인데, 안에서 화면이 바뀌는 앱
- 페이지 이동할 때마다 서버에서 새 HTML을 받아오는 게 아니라, 
앱 내부에서 필요한 화면만 갈아끼우는 방식


## JSP 방식과 React 방식의 흐름 차이

| 구분 | JSP / 전통 서버 렌더링 방식 | React / SPA 방식 |
|---|---|---|
| 주소 예시 | `/care-records/3` | `/care-records/3` |
| 첫 처리 주체 | 서버 | 브라우저 안의 React Router |
| 주소 해석 | 서버가 URL을 보고 어떤 컨트롤러/JSP를 실행할지 결정 | React Router가 URL을 보고 어떤 페이지 컴포넌트를 보여줄지 결정 |
| 페이지 생성 위치 | 서버 | 프론트엔드(브라우저) |
| 처리 흐름 | 브라우저 요청 → 서버 수신 → 컨트롤러 실행 → DB 조회 → JSP 렌더링 → HTML 응답 | 브라우저 주소 변경 → React Router 경로 매칭 → 페이지 컴포넌트 렌더링 → 필요 시 API 호출 → 데이터 받아 화면 갱신 |
| HTML 생성 | 서버가 완성된 HTML을 만들어 응답 | 기본 앱은 이미 로드되어 있고, React가 화면 컴포넌트를 조립해서 표시 |
| DB 접근 | 보통 서버가 직접 DB 조회 후 JSP에 전달 | 페이지 컴포넌트가 백엔드 API를 호출하고, 백엔드가 DB 조회 |
| 화면 전환 방식 | 새 요청 후 페이지 전체가 다시 로드됨 | 보통 전체 새로고침 없이 필요한 화면만 교체 |
| 백엔드 역할 | 화면 생성 + 데이터 처리 | 주로 데이터 제공(API) |
| 프론트 역할 | 서버가 준 HTML 표시 | URL에 맞는 컴포넌트 렌더링 + API 호출 결과로 화면 갱신 |
| 한 줄 요약 | 서버가 페이지를 만들어서 내려줌 | 프론트가 페이지를 그리고, 백엔드는 데이터를 줌 |

## 같은 `/care-records/3` 이지만 내부 흐름은 다름

| 단계 | JSP | React |
|---|---|---|
| 1 | 브라우저가 `/care-records/3` 요청 | 브라우저 주소가 `/care-records/3` 으로 변경됨 |
| 2 | 서버가 요청을 받음 | React Router가 현재 URL을 확인함 |
| 3 | 컨트롤러 실행 | `CareRecordDetailPage` 컴포넌트를 렌더링함 |
| 4 | 서버가 DB 조회 | 페이지 내부에서 필요 시 API 호출 |
| 5 | JSP 렌더링 | 백엔드 API가 데이터 반환 |
| 6 | 완성된 HTML 응답 | React가 데이터를 받아 화면을 다시 그림 |

> 같은 `/path` URL을 사용하더라도,  
> JSP 방식은 **서버가 페이지를 만들어 응답**하고,  
> React 방식은 **프론트가 URL에 맞는 페이지를 렌더링한 뒤 필요한 데이터를 API로 가져와 화면을 구성**
