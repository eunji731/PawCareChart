# HttpOnly

## HttpOnly?
- 쿠키에 붙는 옵션
- 이 옵션이 붙은 쿠키는 자바스크립트에서 읽을 수 없음
- 브라우저는 쿠키를 가지고 있고 서버로 요청할 때 쿠키를 보낼 수 있지만 해당 옵션이 있는 경우 프론트 자바스크립트 코드에서는 읽기 불가
- 인증 쿠키를 자바스크립트에서 직접 읽지 못하게 해서 토큰 탈취 위험을 줄이는 쿠키 옵션

## 사용 이유
- 보안
- 로그인 인증 토큰을 쿠키에 담을 때 중요
- access token이나 refresh token을 브라우저에 저장할 경우
    - localStorage, sessionStorage, 일반 쿠키, HttpOnly 쿠키에 저장 가능
    - localStorage, 일반쿠키는 자바스크립트 접근 가능 -> XSS 공격시 토큰 탈취가능
    - HttpOnly 쿠키는 자바스크립트 접근 불가 -> XSS 공격시 토큰 탈취 방어 가능
- HttpOnly 쿠키는 XSS 공격으로부터 토큰을 보호하는 데 효과적

## 완전보안 가능?
- NO
- HttpOnly는 XSS에 대한 피해를 줄이는 장치 -> 모든 공격 방어 X
- 방어 가능
    - JS 로 쿠키 값 직접 훔치기 어려움
    - document.cookie로 access token 꺼내가기 어려움
- 방어 불가
    - 브라우저가 해당 사이트로 요청 보낼 때 쿠키를 자동 포함하는 것
    - CSRF 같은 공격
     -사이트 내부에서 악성 요청이 실행되는 것 자체
- 쿠키 값을 읽는 것은 방어 가능하지만 브라우저가 쿠키를 실어서 요청 보내는 것 까진 방어 불가

## 실제 흐름
- 1. 로그인 요청 (프론트가 계정정보를 서버에 전달)
- 2. 서버가 쿠키 발급 -> 브라우저는 쿠키를 저장
- 3. 이후 요청 (프론트가 요청을 보내면 브라우저가 쿠키를 자동으로 실어서 보냄)
- 4. 프론트 코드 JS에서는 확인 불가 (HttpOnly)

## 프로젝트에서의 구조
- 로그인 성공시 서버가 HttpOnly 쿠키 발급
- 프론트 -> 토큰 값 직접 설정 X
- 요청 전달 시 브라우저가 쿠키 자동 포함
- 서버가 쿠키 보고 인증 판단
- => 프론트 : 토큰을 들고 다니는 것이 아닌 브라우저가 쿠키를 잘 보내도록 설정하는 역할

## localStorage 방식과 차이
- localStorage : 프론트가 직접 토큰 읽고 헤더에 넣음
    - 장점 : 프론트가 토큰을 직접 제어 가능
    - 단점 : XSS 공격시 토큰 탈취 가능
- HttpOnly 쿠키 : 브라우저가 쿠키를 자동으로 실어서 보냄
    - 장점 : XSS 공격시 토큰 탈취 방어 가능
    - 단점 : 프론트가 토큰을 직접 제어 불가
    - 브라우저와 서버가 쿠키 중심으로 인증처리

## 옵션
- `HttpOnly` : 자바스크립트에서 쿠키 접근 불가
- `Secure` : https 통신일 때만 쿠키 전송
- `SameSite` : 다른 사이트에서 쿠키 전송 제한

## WHY CSRF?
- 브라우저는 쿠키를 자동으로 보냄
- 브라우저가 해당 도메인이면 쿠키를 자동으로 첨부하기 때문에 사용자가 의도하지 않은 요청에도 인증 쿠키 실릴 수 있음
- => CSRF 공격 방어 필요
    - HttpOnly는 XSS 공격으로부터 토큰을 보호하는 데 효과적
    - CSRF 공격 방어에는 CSRF 토큰 사용
