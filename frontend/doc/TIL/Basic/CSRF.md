# CSRF(Cross-Site Request Forgery)

## CSRF?
- 사이트 간 요청 위조
- 사용자가 로그인된 상태라는 점을 악용해서, 다른 사이트가 사용자인 척 요청을 보내게 만드는 공격
- 브라우저가 쿠키를 자동 전송하는 특성을 악용해, 사용자가 의도하지 않은 요청을 보내게 만드는 공격

## CSRF 공격 시나리오
1. 사용자가 A 사이트에 로그인
2. A 사이트가 인증 쿠키를 브라우저에 저장
3. 사용자가 악성 B 사이트에 방문
4. B 사이트가 A 사이트로 위조된 요청 전송
5. 브라우저가 인증 쿠키를 자동으로 첨부
6. A 사이트는 정상 요청으로 판단하고 처리

## WHY 발생?
- 브라우저가 쿠키를 자동으로 첨부하기 때문

## XSS 차이
- XSS : 공격자가 악성 스크립트를 심어서 사용자의 브라우저에서 실행, 스크립트를 훔쳐서 내부에서 공격(내 사이트 안에서 js 실행)
- CSRF : 공격자가 위조된 요청을 보내서 사용자의 브라우저가 서버로 전송, 브라우저의 자동 쿠키 전송을 악용해서 외부에서 요청 위조(다른 사이트를 이용해서 내 사이트로 요청을 보내게 하는 쪽)

## 중요한 이유
- CSRF는 쿠키 기반 인증 구조에서 특히 문제!
- 쿠키는 자동 전송이기 때문
- localStorage는 자동 전송이 아니기 때문에 CSRF 공격에 안전
- 쿠키 인증 → CSRF 위험 큼
- 헤더 기반 Bearer 토큰 → 상대적으로 CSRF 위험 낮음
- => HttpOnly 쿠키 인증 시용시 CSRF 방어 필수

## 방어
- 1. CSRF 토큰 사용
    - 서버가 사용자를 위한 랜덤 토큰을 하나 발급
    - 쿠키에도 저장하거나 HTML/메타태그에 숨겨놓음
    - 요청 시 프론트가 쿠키에서 토큰을 읽어서 요청 헤더에 포함
    - 서버는 요청 헤더의 토큰과 쿠키의 토큰이 일치하는지 확인(쿠키여부, 헤더여부, 일치여부)
- 2. SameSite 쿠키 옵션 사용
    -SameSite를 설정해서 다른 사이트에서 오는 요청에는 쿠키가 덜 실리게 하는 것(브라우저가 쿠키를 자동으로 첨부하는 것을 막는 옵션)
    - `SameSite=Strict` : 모든 사이트에서 쿠키 전송 차단
    - `SameSite=Lax` : GET 요청만 허용
    - `SameSite=None` : 모든 사이트에서 쿠키 전송 허용 대신 `Secure` 필요
    - `Secure` : HTTPS에서만 쿠키 전송
    - 크로스 도메인 프론트/백 분리 환경에서는 NONE 필요경우 존재 -> 그럴수록 CSRF 토큰 방어 중요
- 3. 서버에서 Origin/Referer 검사
    - 서버가 요청 헤더의 Origin이나 Referer를 보고 정상 프론트 도메인에서 온 요청인지 검사
    - 단, Referer는 위조 가능하고, 일부 브라우저/프록시에서 생략될 수 있어서 100% 신뢰는 어려움

## 프로젝트 내 CSRF 방어
```typescript
withXSRFToken: true,
xsrfCookieName: 'XSRF-TOKEN',
xsrfHeaderName: 'X-XSRF-TOKEN',
```
- `xsrfCookieName: 'XSRF-TOKEN'` : 브라우저 쿠키 중 XSRF-TOKEN이라는 이름의 값을 찾겠다
- `xsrfHeaderName: 'X-XSRF-TOKEN'` : 요청 헤더에 X-XSRF-TOKEN이라는 이름으로 토큰을 실어서 보내겠다
- `withXSRFToken: true` : Axios가 이 동작을 자동처리하도록 하겠다.  CSRF 토큰을 사용하겠다

### 흐름
1. 서버가 로그인 처리하면서 CSRF XSRF-TOKEN 쿠키 발급
2. 브라우저 쿠키에 XSRF-TOKEN 저장
3. Axios가 요청 보낼 때 쿠키에서 XSRF-TOKEN 읽어서 헤더에 추가
4. 서버가 헤더의 토큰과 쿠키의 토큰 비교
5. 일치하면 정상 요청으로 처리

## 쿠키만 X, 헤더 토큰도 O
- 브라우저는 쿠키를 자동으로 보낼 수 있음
- 헤더에 CSRF 토큰을 넣는 건 보통 정상 프론트 코드가 해야 하는 동작
- 공격자는 “브라우저가 쿠키를 붙이게 만드는 것”은 노릴 수 있어도
“정상 프론트만 아는 CSRF 토큰을 정확히 헤더에 넣는 것”은 어렵게 만들자는 것
- 그래서 쿠키만 검사하면 부족하고, 쿠키 + 헤더 토큰 같이 확인하는 구조가 의미 O
