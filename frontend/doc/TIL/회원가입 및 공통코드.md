## 1. interface vs type
- `interface`와 `type`은 둘 다 “타입의 모양을 정의”하는 데 사용
- 값의 형태를 설명
- 객체 모양이면 interface를 우선 떠올리고, 그보다 더 복잡한 타입 표현이 필요하면 type을 쓴다
- `interface`
    - 객체 구조 정의에 주로 사용
    - props
    - API 응답 모양
    - 선언 병합 가능
    - 확장 가능
    - extends
- `type`
    - 복잡한 조합 타입
    - 더 넓은 범위에서 사용 가능(값의 범위 표현 가능)
    - Union, Intersection, Tuple 등 표현 가능
    - 선언 병합 불가능(에러)
    - 확장 가능
    - &

```typescript
// interface
const user = {
  id: 1,
  name: '봉봉',
  email: 'test@test.com'
};

interface User {
  id: number;
  name: string;
  email: string;
}

type User = {
  id: number;
  name: string;
  email: string;
};
```
- => 객체 모양만 정의할 때는 interface와 type이 비슷하게 보임

### interface 와 type 핵심차이
- 'interface'
    - 객체 구조를 설명하는 데 최적화된 문법(객체 설계서 느낌)
    - extends로 상속 느낌 구조 만들기 좋음
    - ```typescript
        interface Animal {
            name: string;
        }

        interface Dog extends Animal {
            breed: string;
        }
      ```
    - 선언 병합(declaration merging)이 가능 (같은 이름의 interface를 여러 번 선언하면 합쳐짐)
    - ```typescript
        interface Animal {
            name: string;
        }

        interface Animal {
            age: number;
        }

        const dog: Animal = {
            name: '멍멍',
            age: 3
        };
      ```
- 'type'
    -  객체뿐 아니라 문자열, 숫자, 유니언, 튜플, 함수 타입, 조합 타입까지 훨씬 넓게 표현 가능(타입 별칭 전반)
    - 단순 타입에 이름 붙이기가 가능
    - ```typescript
        type ID = string | number;
        // 유니언 타입 가능
        type Status = 'pending' | 'success' | 'error';
        // 튜플 타입 가능
        type Point = [number, number];
        // 함수 타입 가능
        type Func = (x: number) => number;

        // 교차 타입(&) 조합
        type User = {
            name: string;
        };
        type Admin = {
            role: string;
        };
        type AdminUser = User & Admin;

        // 조건부 타입, mapped type 같은 고급 타입과 궁합 좋음(제네릭/조합형 타입은 type이 훨씬)
        type Filter<T, Condition> = T extends Condition ? T : never;
        type StringOnly = Filter<string | number | boolean, string>;

        // interface 불가
        interface UserName = string;
      ```

### 실무 기준
- 객체면 interface 먼저
- 객체 말고 다른 타입이 섞이면 type

## 2. err:unknown 타입
```typescript
catch (err: unknown) {      console.error('회원가입 에러:', err);}
// catch로 잡힌 에러 변수 err의 타입을 일단 unknown으로 보겠다
```

### unknown
- unknown은 TypeScript에서 “무슨 타입인지 아직 확실하지 않음”을 의미
- any와 비슷하지만 더 안전
- `any` : 문법상 허용하여 런타임에 에러 발생가능
- `unknown` : 문법상 제한하여 런타임 에러 방지
- any는 타입 검사를 아예 안 하지만(막써도 됨) unknown은 “이 타입이 뭔지 확실히 밝혀지기 전까지는 함부로 쓰지 마”라고 막아줌
- unknown을 쓰면 타입스크립트는 “이 변수에는 어떤 타입이든 들어올 수 있지만, 그 타입을 쓰려면 네가 직접 타입을 확인하고 안전한 타입으로 바꿔줘야 해”라고 강제 -> 덕분에 런타임 에러를 컴파일 타임에 잡을 수 있음

### catch에서 unknown 사용
- catch로 잡히는 에러는 항상 같은 모양이 아님
- 진짜 Error 객체일 수도 있고 문자열일 수도 있고 숫자일 수도 있고 axios 에러 객체일 수도 있고 이상한 객체도 가능
- TypeScript 입장에서는 “무슨 타입인지 확신 못 하겠다” => `unknown`

### 프로젝트 코드 예시
```typescript
} catch (err: unknown) {
  console.error('회원가입 에러:', err);

  const errorResponse = (err as { response?: { data?: { message?: string } } })?.response;
  const errorMessage =
    errorResponse?.data?.message ||
    '회원가입 중 오류가 발생했습니다. 잠시 후 다시 시도해주세요.';

  setError(errorMessage);
}
```
- 1. err가 뭔지 정확히 모름 -> unknown
- 2. 하지만 이 코드는 “아마 axios 에러일 것”이라고 기대
- 3. 그래서 err.response?.data?.message를 시도 (확실하진 않지만, response.data.message 같은 구조일 수도 있다고 보고 접근)
- 4. 만약 에러가 발생하면 undefined 반환
- 5. undefined면 "회원가입 중 오류가 발생했습니다. 잠시 후 다시 시도해주세요."로 설정


### 사용
- 1. 타입 좁히기
    - ```typescript
      function printError(err: unknown) {
        if (err instanceof Error) {
          console.log(err.message);
        } else {
          console.log('알 수 없는 에러:', err);
        }
      // err가 진짜 Error면 message 사용, 아니면 다른 처리
      ```
- 2. 타입 단언
    - ```typescript
      const errorResponse = (err as { response?: { data?: { message?: string } } })?.response;
      // 지금은 이 값이 이런 구조라고 내가 믿고 쓸게
      ```
- 3. 라이브러리 전용 타입 가드 사용
    - ```typescript
        import axios from 'axios';

        catch (err: unknown) {
        if (axios.isAxiosError(err)) {
            console.log(err.response?.data);
            }
        }
        // 정석적인 방식
        // 진짜 axios 에러인지 확인하고 사용하는 것
      ```

### unknown vs any
- `any`와 `unknown`은 둘 다 “정확한 타입을 모른다”는 점은 같지만, TypeScript가 대하는 태도가 완전히 다름
- `any`
    - 모르겠으니까 그냥 네 마음대로 써(검사 포기)
    - 컴파일 단계에서는 에러를 안 냄
    - 런타임 에러 가능
- `unknown`
    - 모르겠으니까 확인하기 전엔 쓰지 마(검사 강제)
    - 컴파일 단계에서 에러를 냄
    - 런타임 에러 방지
- TypeScript는 실행 전에 코드를 검사하는 도구 -> 모르는데도 그냥 통과시킬 거냐(any), 모르니까 막고 확인을 요구할 거냐(unknown)
- ```typescript
    // any
    let value: any = 10;
    value = 'hello'; // 허용
    value.toUpperCase(); // 런타임 에러 가능

    // unknown
    let value: unknown = 10;
    value = 'hello'; // 허용
    value.toUpperCase(); // 컴파일 에러!

    // unknown을 안전하게 사용하려면
    function printValue(value: unknown) {
      if (typeof value === 'string') {
        console.log(value.toUpperCase()); // 안전
      }
    }
  ```

### 런타임 vs 컴파일타임
- `런타임` : 실제 브라우저에서 코드가 실행되는 시점
- `컴파일타임` : TypeScript가 코드 검사하는 시점

## 3. why useEffect안에서는 바로 await 사용이 안되지?

### useEffect
- useEffect는 렌더링 후 실행할 부수효과를 등록하고, 그 effect를 나중에 해제할 방법까지 React가 관리하려는 훅이기 때문
- "비동기 함수 실행 도구"가 아니라, "렌더링이 끝난 뒤 부수효과를 등록하고, 필요하면 나중에 정리(cleanup)까지 연결하는 장치"
- 따라서 아무것도 반환하지 않기 (void), 정리 함수 반환하기 (cleanup function)만 가능
- 리액트는 useEffect를 등록 + 해제 구조로 다루기 때문

- Promise는 React가 써먹을 수 있는 반환값이 아니어서 원하지 않는 것
    - React는 effect 반환값을 보고 “이걸 나중에 실행해서 정리하면 되나?”를 판단 시도
    - Promise :  나중에 끝날 수도 있고, 비동기 결과를 담고 있을 뿐이고, “정리 함수”가 아님
    - 즉 React 입장에서는 Promise를 받아도 할 수 있는 게 없음

- React는 effect를 단발성 작업으로만 보지 않음 -> 시작한 건 나중에 정리 가능하다는 전제
- effect는 “실행”만 중요한 게 아니라 생명주기 끝에서 정리할 수 있어야 함
    - 이벤트 리스너(등록했으면 해제해야 함), setInterval(시작했으면 clearInterval 해야 함), 웹소켓(연결했으면 닫아야 함), 구독(subscription)(
    구독했으면 unsubscribe 해야 함)

### 부수효과
- 화면을 그리는 일 자체 말고, 화면 바깥 세계와 상호작용하는 작업
- API 호출, 이벤트 리스너 등록, 타이머 시작, 웹소켓 연결, 외부 라이브러리 초기화, DOM 직접 건드리기
- 리액트는 렌더링과 부수효과를 분리
- 순수하게 렌더링을 하고 이펙트를 실행, 다음 이펙트 전에 이전 이팩트 정리

### why?
- useEffect가 받는 콜백 함수는 async 함수가 되면 안 되기 때문
- useEffect가 async를 써야 하는 게 아님
- effect 안에서 하려는 작업 중에 비동기 작업이 자주 있을 뿐
- useEffect의 목적은 “비동기 처리”가 아니라 “렌더링 후 부수효과 실행”
- useEffect 안에서 async가 자주 등장할 뿐, useEffect 자체가 async를 요구하는 건 아님


### await는 어디에서 사용가능?
- await는 기본적으로 async 함수 안에서만 사용 가능
- async가 아니면 await 사용 불가


### useEffect가 원하는 함수
- 아무것도 반환 X
- 정리 함수(cleanup)를 반환

### 정리 함수(cleanup)
- useEffect가 반환하는 함수
- 컴포넌트가 언마운트되거나, effect가 다시 실행되기 직전에 호출
- 타이머 해제, 이벤트 리스너 제거 등 정리 작업 수행.
- `useEffect` 실행 → 무언가 설치함
- `cleanup` 함수 → 그 설치한 것을 치움
- 기본 문법
    ```typescript
    // 기본 문법
    useEffect(() => {
    // 실행할 작업

    return () => {
        // 정리할 작업
    };
    }, [의존성]);
    ```
- 예시
    ```typescript
    import { useEffect, useState } from 'react';

    function TimerBox() {
    useEffect(() => {
        console.log('이펙트 실행: 타이머 시작');

        const intervalId = setInterval(() => {
        console.log('1초마다 실행 중...');
        }, 1000);

        return () => {
        console.log('클린업 실행: 타이머 정리');
        clearInterval(intervalId);
        };
    }, []);

    return (
        <div style={{ padding: '12px', border: '1px solid #ccc', marginTop: '12px' }}>
        타이머 켜짐
        </div>
    );
    }

    export default function EffectCleanupExample() {
    const [isVisible, setIsVisible] = useState(false);

    const handleToggle = () => {
        setIsVisible((prev) => !prev);
    };

    return (
        <div style={{ padding: '20px' }}>
        <h2>useEffect cleanup 예제</h2>

        <button onClick={handleToggle}>
            {isVisible ? '타이머 끄기' : '타이머 켜기'}
        </button>

        <p style={{ marginTop: '12px' }}>
            현재 상태: {isVisible ? '켜짐' : '꺼짐'}
        </p>

        {isVisible ? (
            <TimerBox />
        ) : (
            <div style={{ marginTop: '12px' }}>꺼졌음</div>
        )}
        </div>
    );
    }
    ```

    - `처음 화면` : isVisible = false, TimerBox 없음, 타이머 없음
    - `버튼 눌러서 켬` : isVisible = true, TimerBox 마운트, useEffect 실행, setInterval 시작, 1초마다 콘솔 출력
    - `버튼 눌러서 끔` : isVisible = false, TimerBox 언마운트, cleanup 실행, clearInterval(intervalId), 콘솔 출력 멈춤

    - `클린업 코드가 없는 경우`
        - 버튼 눌러서 꺼졌음이 화면에 떠도 콘솔에서는 계속 로그 표출
        - 화면에서는 꺼진 것처럼 보임 -> 하지만 실제로는 타이머가 살아있음

    - 예제 소스 구조
        - 버튼 클릭
        -> 상태 변경
        -> 컴포넌트 사라짐
        -> cleanup 실행
        -> 타이머 정리


### async 함수의 반환
- `async` 함수는 무조건 `Promise`를 반환

### 리액트가 원하는 흐름
- 1단계 -> 컴포넌트 렌더링 완료
- 2단계 -> effect 실행
- 3단계 -> 필요하면 cleanup 함수 저장
- 4단계 -> 다음 effect 실행 전 또는 unmount 시 cleanup 실행

## 캐시방지파라미터

## why? 이펙트 안에 함수를 만들고 그걸 호출? 밖에서 만들고 이펙트를 깔끔하게 만들면?
- 핵심은 “그 함수가 어떤 값을 써야 하느냐”, “언제 새로 만들어져야 하느냐”
- 이펙트 안에 함수 만드는 이유
    - 그 effect 안에서만 쓰는 함수라서
    - 현재 시점의 state, props 값을 자연스럽게 잡아야 해서
    - 의존성 관리가 더 직관적이라서
- 밖으로 빼도 되는 경우
    - 컴포넌트 state나 props를 안 쓰는 순수 함수
    - 여러 곳에서 재사용할 함수
    - effect와 상관없는 공통 로직

### effect는 언제 쓰는 거냐
- 컴포넌트가 나타난 뒤 자동으로 무언가 해야 할 때
- 특정 값이 바뀌면 다시 실행해야 할 때
- 외부 시스템과 연결해야 할 때
- (초기 데이터 조회, props 바뀔 때 재조회, 이벤트 리스너 등록, 타이머 시작)

- 안쓰는 경우
- (버튼 클릭, 폼 제출, 사용자가 직접 발생시킨 이벤트 응답)
