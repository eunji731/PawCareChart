# 리액트는 DOM을 직접 안 만지고 어떻게 화면을 바꾸는가

## 방식
- 예전 방식: 개발자가 DOM 직접 조작
- 리액트 방식: 개발자가 상태(state) 와 보여줄 화면 모양을 선언 -> 리액트가 DOM 변경 반영

### 예시
#### 순수 JS
1. DOM 요소 찾기
2. 클릭 이벤트 연결
3. 값 바꾸기
4. 다시 DOM 텍스트 바꾸기

#### 리액트
- 개발자의 일
    - 상태를 정의
    - 보여주라고 선언
    - 특정 조건일 때 상태를 바꾸는 로직 작성
- 리액트가 알아서 할 일
    - 상태가 바뀌면 화면을 다시 계산(변경되었는지 감지 후 화면 계산)
    - 실제 DOM을 효율적으로 업데이트

### 리액트
- 상태가 바뀌면 화면이 다시 그려진다
- 상태(state)가 바뀌면 컴포넌트가 다시 실행되고, 결과 화면이 다시 계산

#### `컴포넌트가 다시 실행된다`
- 리액트 컴포넌트는 보통 함수
- 상태가 바뀌면 리액트는 이 함수를 다시 실행해서 “지금 상태 기준으로 화면이 어떻게 생겨야 하지?”를 다시 계산
- 함수가 다시 호출되어 새 JSX를 돌려주는 것
- 리렌더링(render again)

#### 매번 다 다시 그리면 비효율?
- 리액트는 내부적으로 필요한 부분만 DOM에 반영
- 리액트는 실제 DOM을 가능한 효율적으로 업데이트하려고 함

#### JSX
```jsx
return <h1>{title}</h1>;
```
- 직접 DOM 조작 코드 X
- 현재 상태 기준으로 h1 안에 title을 보여달라는 선언
- 리액트는 이 선언을 보고 필요하면 DOM을 바꾸고, 필요 없으면 안 바꿈
- JSX는 화면의 모양을 선언하는 문법이지, DOM 직접 조작 명령문 X

### 순수 JS vs React
#### 순수 JS
1. DOM 요소 찾기
2. 이벤트 연결
3. 값 변경
4. DOM 다시 수정
=> 개발자가 직접 화면을 만짐

#### React
1. 상태 정의
2. 상태에 따라 어떤 화면이 보여야 하는지 선언
3. 상태 변경 함수 호출
4. DOM 반영은 리액트가 처리
=> 개발자는 화면 규칙을 적고, 리액트가 실제 변경 작업을 처리
=> DOM을 직접 조작하는 대신, 상태가 바뀌면 화면을 다시 계산해서 필요한 DOM 변경을 대신 처리해주는 방식

#### 상태(state)
- 리액트에서 상태(state)는 화면을 결정하는 데이터
- (입력창 값 , 로그인 여부, 목록 데이터, 선택된 탭, 모달 열림 여부, 로딩 여부) 등
- 상태가 바뀌면 리액트는 자동으로 화면을 다시 그림
- DOM을 직접 조작하는 대신, 상태를 바꿔서 화면을 바꾸는 구조
- 직접 DOM 조작 대신 상태 변경이 정석
