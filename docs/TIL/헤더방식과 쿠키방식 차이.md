# JWT 헤더 방식 vs JWT 쿠키 방식 비교 정리

## 1. 전제
두 방식 모두 JWT를 사용할 수 있다. 차이는 "인증 토큰을 어디에 두고, 누가 요청에 실어 보내며, 서버는 어디서 읽느냐"에 있다.

- 헤더 방식: 프론트가 토큰을 저장하고 `Authorization: Bearer ...` 헤더에 실어 보낸다.
- 쿠키 방식: 서버가 `Set-Cookie`로 JWT를 저장시키고, 브라우저가 이후 요청에 자동으로 쿠키를 붙인다.

---

## 2. 가장 큰 구조 차이

### 2.1 헤더 방식
로그인 후:
1. 서버가 access token을 응답 body로 반환
2. 프론트가 토큰을 저장
3. 이후 요청마다 프론트가 Authorization 헤더를 직접 붙임

예:
```http
Authorization: Bearer eyJhbGci...
```
### 2.2 쿠키 방식

로그인 후:

서버가 Set-Cookie로 access token을 쿠키에 저장시킴
브라우저가 쿠키를 저장
이후 요청마다 브라우저가 조건에 맞으면 쿠키를 자동 전송

예:

Cookie: accessToken=...

서버는 요청 헤더 Authorization가 아니라 쿠키를 읽어 JWT를 검증한다.

## 3. 프론트 관점 차이
### 3.1 헤더 방식에서 프론트 역할
로그인 응답에서 토큰을 꺼냄
localStorage, sessionStorage, 메모리 등 어디엔가 저장
axios interceptor / fetch wrapper로 매 요청마다 헤더 삽입
만료 시 갱신 로직 구현

즉 프론트가 토큰을 직접 관리한다.

### 3.2 쿠키 방식에서 프론트 역할
로그인 응답의 Set-Cookie는 브라우저가 처리
이후 요청 시 credentials: 'include' 또는 withCredentials: true 같은 설정 필요
Authorization 헤더 직접 삽입 로직은 줄어듦
대신 로그인 여부 확인용 /me 호출 등에 의존하는 패턴이 많아짐

즉 토큰 문자열 자체를 프론트가 직접 다루는 책임이 줄어든다.

## 4. 백엔드 관점 차이
### 4.1 헤더 방식 백엔드

서버는 보통 이렇게 처리한다.

Authorization 헤더 읽기
Bearer 제거
JWT 검증
인증 객체 생성
SecurityContext 저장

백엔드는 헤더 파싱 로직이 핵심이다.

### 4.2 쿠키 방식 백엔드

서버는 보통 이렇게 처리한다.

로그인 시 Set-Cookie로 JWT 저장
요청 시 request.getCookies()로 쿠키 읽기
accessToken 추출
JWT 검증
인증 객체 생성
SecurityContext 저장

백엔드는 쿠키 발급 정책까지 책임져야 한다. Spring의 ResponseCookieBuilder는 sameSite, secure, httpOnly 등을 지원한다.

## 5. 보안 차이
### 5.1 헤더 방식의 장점

헤더는 브라우저가 자동으로 붙이는 값이 아니다. 보통 자바스크립트 코드가 명시적으로 넣는다. 그래서 쿠키 자동 전송을 악용하는 형태의 CSRF와는 결이 다르다. 즉 CSRF 부담은 상대적으로 작다. OWASP가 설명하는 CSRF의 핵심은 브라우저가 자동으로 인증 정보를 보내는 특성이다.

### 5.2 헤더 방식의 약점

토큰을 프론트가 보관한다. localStorage 같은 곳에 저장하면 자바스크립트로 접근 가능한 경우가 많으므로 XSS 상황에서 토큰이 탈취될 위험이 더 커질 수 있다. 반면 HttpOnly 쿠키는 JS 접근을 제한한다.

### 5.3 쿠키 방식의 장점
HttpOnly로 토큰 노출을 줄일 수 있다.
Secure, SameSite 같은 브라우저 정책을 활용할 수 있다.
### 5.4 쿠키 방식의 약점
쿠키 자동 전송 때문에 CSRF를 반드시 고려해야 한다.
SameSite, Secure, CORS, credentials 설정이 조금만 어긋나도 인증이 깨질 수 있다.
## 6. 개발 난이도 차이
### 6.1 헤더 방식

설정 포인트:

로그인 응답 처리
토큰 저장
인터셉터/공통 fetch wrapper
401 처리
refresh 재시도

즉 프론트 로직 비중이 크다.

### 6.2 쿠키 방식

설정 포인트:

로그인 시 Set-Cookie 발급
쿠키 속성 설계(HttpOnly, Secure, SameSite, Path, Max-Age)
CORS + credentials
프론트의 withCredentials / credentials: 'include'
refresh / logout 쿠키 일관성

즉 프론트 토큰 관리 코드는 줄지만, 프론트-백엔드-브라우저 정책을 함께 맞춰야 한다.

## 7. 저장 위치 차이
### 7.1 헤더 방식

토큰 저장 위치는 보통:

localStorage
sessionStorage
메모리 상태
중 하나다.
### 7.2 쿠키 방식

토큰 저장 위치는 브라우저 쿠키 저장소다. 다만 HttpOnly 쿠키는 자바스크립트에서 직접 읽기 어렵다.

## 8. 인증 실패 양상 차이
### 8.1 헤더 방식에서 흔한 오류
토큰 저장 누락
인터셉터 누락
Bearer 접두사 누락
만료 토큰 재시도 로직 실패
### 8.2 쿠키 방식에서 흔한 오류
Set-Cookie는 내려갔지만 브라우저가 저장하지 않음
저장은 됐지만 다음 요청에 안 붙음
SameSite=Strict 때문에 특정 유입 흐름에서 안 붙음
Secure=true인데 HTTPS가 아니라 안 붙음
CORS credentials 설정 불일치
프론트가 credentials: 'include'를 빼먹음

즉 헤더 방식은 코드 누락 오류가 많고, 쿠키 방식은 브라우저 정책 오류가 많다.

## 9. refresh / logout 차이
### 9.1 헤더 방식
refresh token도 프론트가 직접 저장하는 경우가 많다
로그아웃은 저장소에서 토큰 제거가 핵심이다
### 9.2 쿠키 방식
access token뿐 아니라 refresh token도 쿠키에 둘지, 일부만 쿠키로 둘지 정책이 중요하다
로그아웃은 서버 응답으로 쿠키 만료(Max-Age=0)를 내려 브라우저에게 삭제를 지시해야 한다
발급 시와 삭제 시의 path, sameSite, secure 조건 일치가 중요하다. Spring ResponseCookieBuilder는 이런 속성을 함께 제어할 수 있다.
## 10. 실무 선택 기준
### 헤더 방식이 잘 맞는 경우
프론트가 인증 로직을 강하게 통제하고 싶을 때
API 클라이언트가 브라우저뿐 아니라 다양한 환경일 때
쿠키/CORS 정책보다 명시적 헤더 방식을 선호할 때
### 쿠키 방식이 잘 맞는 경우
웹 브라우저 기반 서비스일 때
access token을 JS에서 직접 다루지 않게 하고 싶을 때
HttpOnly + Secure + SameSite 조합을 활용하고 싶을 때
프론트의 토큰 저장 책임을 줄이고 싶을 때

단, 쿠키 방식은 CSRF와 브라우저 정책을 반드시 함께 설계해야 한다.

## 11. 핵심 비교 요약
### 헤더 방식
토큰 저장 주체: 프론트
전송 주체: 프론트 코드
서버가 읽는 위치: Authorization 헤더
강점: CSRF 부담 상대적으로 작음
약점: XSS 시 토큰 노출 위험이 상대적으로 큼
### 쿠키 방식
토큰 저장 주체: 브라우저 쿠키
전송 주체: 브라우저
서버가 읽는 위치: Cookie
강점: HttpOnly / Secure / SameSite 활용 가능
약점: CSRF, SameSite, CORS, credentials 설계를 함께 맞춰야 함
## 12. 결론

두 방식 모두 JWT 인증이라는 점은 같다. 차이는 "토큰을 누가 보관하고, 누가 보내며, 서버가 어디서 읽느냐"에 있다.

### 헤더 방식은 프론트 중심
### 쿠키 방식은 브라우저 + 백엔드 정책 중심

보안 관점에서는 보통 다음처럼 이해하면 된다.

### 헤더 방식: CSRF 부담은 낮지만 XSS로 토큰이 노출될 위험을 더 의식해야 한다
### 쿠키 방식: HttpOnly로 XSS 노출 위험을 줄이는 데 도움을 받을 수 있지만, 자동 전송 특성 때문에 CSRF와 SameSite 설계를 더 신경 써야 한다.
